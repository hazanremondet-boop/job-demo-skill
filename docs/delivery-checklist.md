# Demo Delivery Checklist

这份清单用于判断一个岗位定制 demo 是否已经可以拿给 HR 或面试官看。

## 1. 岗位命中

- 项目名称能让人看出业务场景。
- 一句话定位包含“给谁用、解决什么问题、输出什么结果”。
- README 或展示材料里明确写出对应的岗位 JD 关键词。
- 至少有 3 个功能能对应岗位要求。
- 没有为了炫技加入和岗位无关的大功能。

## 2. 产品闭环

- 有真实输入，而不是只有静态展示。
- 有处理过程，比如 AI 分析、数据计算、状态流转或检索。
- 有明确输出，比如报告、评分、图表、建议、文案或记录。
- 用户能完成一个从输入到结果的主流程。
- 空状态、加载状态、失败状态至少有基本处理。

## 3. 工程完整度

- 有清晰页面结构。
- 有接口或本地处理逻辑。
- 有基本数据结构。
- 有 mock 数据或示例数据。
- 有可复制、导出或保存结果的能力。
- 代码能在本地按 README 跑起来。

## 4. AI 应用质量

- AI 不是单纯聊天框，而是嵌入业务流程。
- Prompt 有角色、任务、输入变量和输出格式。
- 输出尽量结构化，便于前端展示。
- 结果有解释，不只给一个分数或结论。
- 对错误输入或空输入有处理。

## 5. 展示材料

- README 包含项目定位、功能、技术栈和运行方式。
- 有 3-5 张截图：输入、处理中、结果、历史/导出、移动端或异常状态。
- 有 90 秒演示脚本。
- 有简历 bullet。
- 有 HR 开场话术。
- 有项目链接、GitHub 链接或录屏链接。

## 6. 面试可讲性

- 能讲清楚为什么选这个场景。
- 能讲清楚为什么删掉一些功能。
- 能讲清楚数据结构和接口设计。
- 能讲清楚 AI prompt 如何迭代。
- 能讲清楚下一步如果继续做，会优化什么。

## 7. 最小合格标准

如果时间很紧，一个 demo 至少要满足：

- 一个可操作页面。
- 一个完整输入到输出流程。
- 一个贴近岗位的业务场景。
- 一段可读的项目说明。
- 一条能放进简历的 bullet。
- 一段能发给 HR 的短消息。

## 8. 高分标准

如果想让它明显超过普通学生项目，继续补：

- 在线部署。
- 历史记录。
- 导出报告。
- 示例数据一键填入。
- 结构化 AI 输出。
- 错误状态和边界处理。
- 简洁但专业的 UI。
- 和目标公司业务强相关的案例数据。
