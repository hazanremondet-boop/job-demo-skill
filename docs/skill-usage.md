# Job Demo Skill Usage

这个 skill 的新口径是：

> 先根据岗位和个人背景生成三条项目路线，帮助用户确认项目内容、产品逻辑和技术栈；用户确认后，再让 Codex 快速生成完整项目。

它不应该被当成单纯的“项目方案生成器”。它应该像一个项目共创助手，先帮用户做选择，再帮用户把选择落地成作品。

## Skill 位置

```text
C:\Users\Administrator\.codex\skills\jd-to-demo-workflow
```

调用名：

```text
$jd-to-demo-workflow
```

## 推荐调用方式

第一步，让 skill 先分析岗位并给出三条路线：

```text
Use $jd-to-demo-workflow.

这是一个 BOSS 直聘岗位截图/JD。请结合我的个人背景，先不要直接生成项目代码。

请先输出三条项目路线：
1. 最快落地路线
2. 岗位命中路线
3. 技术展示路线

每条路线都要说明：
- 项目内容
- 产品逻辑和核心用户流程
- 为什么匹配这个岗位
- 推荐技术栈和原因
- MVP 页面、功能、接口和数据结构
- 预计开发时间
- 风险和降级方案
- 简历 bullet 和 HR 话术角度

最后请推荐最适合我的一条路线，并等我确认后再进入项目生成。

我的背景：
- 大二/大三/研一
- 目标岗位：
- 已有技术栈：
- 已有项目：
- 可投入时间：
- 偏好：快速落地 / 岗位命中 / 技术展示 / 不确定
```

第二步，用户确认路线后，让 Codex 直接生成项目：

```text
我确认选择路线 B：岗位命中路线。

请基于这条路线直接进入项目生成阶段，产出一个可以本地运行的完整项目。

要求：
- 生成完整代码
- 提供 README
- 提供 .env.example
- 支持 mock 模式
- 不要把 API key 写进前端
- 提供示例数据
- 提供验收标准
- 提供简历 bullet
- 提供 HR/BOSS 话术
- 提供 90 秒演示脚本
```

## Skill 应该如何输出

在用户还没有确认路线前，输出重点是路线比较，不是完整代码。

建议结构：

1. 岗位信息识别
2. 招聘需求理解
3. 个人 fit/gap 分析
4. 三条项目路线
5. 路线对比表
6. 推荐路线
7. 等待用户确认

在用户确认路线后，输出重点是项目生成。

建议结构：

1. 已确认路线
2. 项目目标和边界
3. 技术栈和目录结构
4. 页面、接口、数据结构和 AI 行为
5. 实现步骤或直接生成代码
6. README、mock 数据和环境变量
7. 验收标准
8. 简历 bullet、HR 话术和演示脚本

## 三条路线的差异

### 最快落地路线

优先保证 1-2 天内能做完。

适合：

- 用户时间紧。
- 技术栈还不完整。
- 需要尽快给 HR 一个可打开的作品。

### 岗位命中路线

优先保证项目和岗位强相关。

适合：

- 用户非常想冲某个岗位。
- 公司业务和 JD 场景比较明确。
- 需要项目一眼看出“就是为这个岗位做的”。

### 技术展示路线

优先保证面试可讲的技术深度。

适合：

- 用户希望展示前后端、数据库、AI API、部署等工程能力。
- 岗位强调技术栈和项目经验。
- 用户有足够时间完成稍复杂的版本。

## 为什么要先确认路线

直接生成项目容易出现三个问题：

- 项目看起来完整，但和岗位关系不够强。
- 技术栈堆得太多，用户讲不清楚。
- 项目范围过大，最后做不完。

先比较三条路线，可以让用户在“快、准、深”之间做选择。确认后再生成项目，成功率更高，也更容易写进简历和发给 HR。

## API key 规则

无论选择哪条路线，都不要把 API key 写进前端代码。

推荐：

- `.env` / `.env.local`
- `.env.example`
- 后端接口或 serverless route 读取环境变量
- mock 模式保证没有 API key 时也能演示

## 好的最终结果

确认路线后的项目应该满足：

- 本地能运行。
- README 能让别人快速启动。
- 页面能展示核心流程。
- mock 数据能支撑演示。
- 技术栈和岗位要求有对应关系。
- 简历 bullet 具体可信。
- HR 话术能自然引出项目链接。
