# JD-to-Demo Skill Usage

这个项目当前优先把能力沉淀到 Codex skill，而不是网页 API。新目标是做一个“招聘需求 + 个人情况 -> 强相关求职项目”的快速项目生成 skill，帮助缺少相关实习/项目经历的大学生用 1-3 天做出可展示 demo。

原因：岗位截图分析、招聘要求理解、求职者诉求匹配、项目生成提示词这些能力更适合先由 skill 承担；网页如果直接接真实 LLM API，会涉及 API key 暴露、后端代理、部署环境和费用控制，当前不是第一优先级。

## Skill 位置

```text
C:\Users\Administrator\.codex\skills\jd-to-demo-workflow
```

调用名：

```text
$jd-to-demo-workflow
```

## 最推荐的输入格式

把岗位截图发给 Codex 后，用这段话：

```text
Use $jd-to-demo-workflow.

这是一个 BOSS 直聘岗位截图。请先识别截图里的岗位信息，再结合我的背景和诉求，生成一份可以直接复制到新 Codex 线程里的完整项目生成提示词。

我的背景：
- 大二
- 正在找 AI 应用 / 全栈工程师实习
- 会 React、Node.js、Python，了解 LLM API
- 没有正式实习经历，但希望用项目 demo 证明能力

我的诉求：
- 项目最好 48 小时内能完成
- 能拿给 HR 看
- 能写进简历
- 要有页面、API、数据结构、AI prompt、mock 模式和 README
- 不要把 API key 写进前端
```

## Skill 应该输出什么

每次调用都应该尽量输出这 6 部分：

1. 岗位信息识别：截图里看得到的事实、推断、不确定项。
2. 招聘需求理解：硬技能、隐藏能力信号、优先级和最适合的证明材料。
3. 求职者画像与诉求：已有能力、需要证明的能力、时间限制，以及和岗位的 fit/gap。
4. 项目策略：推荐做什么 demo，为什么适配岗位，哪些范围明确不做。
5. 完整 Codex 项目生成提示词：给另一个 Codex 直接开发项目用。
6. 投递材料：简历 bullet、HR 消息、必要时加 90 秒演示脚本。

## 为什么这比网页 API 更优先

网页工具适合做可视化流程，但真正难的是“把岗位和个人诉求压缩成一个高质量项目任务书”。这个任务书一旦生成，就可以交给 Codex 开发项目。

所以第一阶段优先级是：

1. 做好 skill。
2. 用 skill 稳定生成项目提示词。
3. 拿提示词让 Codex 生成真实 demo。
4. 再考虑把流程产品化成网页工具。
